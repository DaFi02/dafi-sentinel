# Exploration: DAFI Sentinel

## Current State

The workspace is a lightweight multi-project repository named `projects`. The root contains a minimal Python 3.13 `uv` package skeleton with no dependencies, an empty `README.md`, and an empty `dafi_sentinel/` directory reserved for the new project. OpenSpec is initialized under `openspec/`, strict TDD is enabled, and the preferred local container runtime is Podman. No product code, tests, or domain specs exist yet for DAFI Sentinel.

## Positioning

DAFI Sentinel should be presented as a portfolio-grade, open-source AI Engineering project by Diego Andía Fernández / DAFI / DaFi02: an AI-assisted incident investigation workbench that turns logs, metrics-like event tables, alerts, and deployment context into explainable root-cause hypotheses.

The strongest GitHub positioning is:

- **AI Engineering, not chatbot glue**: stateful investigation workflows, tool boundaries, audit trails, evidence handling, and testable ML components.
- **Operations domain credibility**: focused on incident triage, timeline reconstruction, anomaly detection, and root-cause analysis.
- **Security-first agent design**: prompt injection defense, sensitive data redaction, scoped tool permissions, human approvals, and auditable reasoning are first-class differentiators.
- **Self-contained product dashboard**: charts and investigation outputs live in DAFI Sentinel itself; Grafana and Prometheus are intentionally out of scope.

## Affected Areas

- `README.md` — currently empty; should become the GitHub-facing project pitch and quickstart.

- `openspec/changes/dafi-sentinel/` — active SDD change folder for this concept.

## Scope Boundaries

### V1

- Ingest sample incident datasets from files or local seeded fixtures: logs, alert events, deployment markers, and simple service metadata.
- Run deterministic preprocessing: parsing, normalization, timeline building, service grouping, redaction, and evidence indexing.
- Use scikit-learn for practical, explainable ML: anomaly scoring, log clustering, incident similarity, and simple ranking features.
- Provide a local API and product dashboard with investigation sessions, hypothesis timelines, evidence cards, and generated charts.
- Support user interaction during investigation: ask clarifying questions, request explanations, inspect evidence, and request charts generated by controlled Python tooling.
- Implement a strong Security Agent layer: prompt injection checks, sensitive data redaction, tool permission boundaries, audit logs, and human approval gates for risky actions.
- Use Podman-compatible local infrastructure for optional services such as Postgres, Redis, or object storage.

### Later Versions

- Live integrations with Kubernetes, cloud logging APIs, SIEM systems, ticketing systems, or deployment platforms.
- Real-time streaming ingestion and alert correlation across many tenants.
- Automated remediation actions beyond safe draft recommendations.
- Advanced model serving, fine-tuned models, vector databases, distributed tracing backends, and production SSO/RBAC.
- Team collaboration features, incident war rooms, and long-term incident knowledge graphs.

## Approaches

1. **LangGraph-centered investigation workflow** — model each incident investigation as a stateful graph with nodes for ingest, redaction, anomaly detection, hypothesis generation, user Q&A, visualization, and approval gates.
   - Pros: aligns with durable execution, persistence, streaming, and human-in-the-loop needs; makes agent state inspectable and testable; supports interruptions for approvals.
   - Cons: adds framework complexity before the domain model is mature; requires disciplined boundaries to avoid graph sprawl.
   - Effort: Medium

2. **Plain service orchestration first** — implement deterministic Python services and add LangGraph later once workflows stabilize.
   - Pros: simplest V1 foundation; lower dependency and learning overhead; easier unit testing at the start.
   - Cons: human-in-the-loop state, resumability, and streaming would need custom design or later migration.
   - Effort: Low

3. **Dashboard-only analytics app** — focus on ingestion, scikit-learn analysis, and charts before agentic workflows.
   - Pros: fastest visual portfolio result; reduces LLM risk.
   - Cons: misses the AI Engineering differentiator and weakens the Security Agent story.
   - Effort: Low

## Recommendation

Use a **hybrid V1**: build deterministic core services first, then wrap the investigation flow in a small LangGraph graph once the first workflow is testable. Keep LangGraph scoped to orchestration and human-in-the-loop state, not business logic.

Recommended high-level architecture:

```text
Product Dashboard
  ├─ Investigation timeline, evidence cards, hypotheses, charts, audit log
  └─ Interactive Q&A and chart requests

API Layer
  ├─ Incident sessions
  ├─ Dataset ingestion
  ├─ Investigation events
  └─ Chart/artifact retrieval

Agent Orchestration
  ├─ Investigation Graph (LangGraph candidate)
  ├─ Security Agent / policy gate
  ├─ Analyst Agent for explanations and hypotheses
  └─ Visualization Tool node using controlled Python execution

Core Domain Services
  ├─ Log parser and normalizer
  ├─ Timeline builder
  ├─ Evidence store
  ├─ Redaction service
  ├─ Anomaly detection and clustering services
  └─ Hypothesis ranking

Local Infrastructure
  ├─ Podman-compatible compose file
  ├─ Postgres or SQLite-first persistence
  ├─ Optional Redis for background jobs
  └─ Local seeded datasets
```

Avoid Grafana and Prometheus entirely in V1. The product dashboard should own charts, summaries, evidence views, and investigation state. If metric-like inputs are needed, treat them as imported datasets rather than external monitoring dependencies.

## Agent Interaction Model

An investigation session should be conversational but evidence-grounded:

- The user opens or creates an incident investigation.
- The system ingests local sample data and builds a normalized timeline.
- The Security Agent redacts sensitive content and classifies tool permissions before agent reasoning.
- The investigation workflow generates initial findings: anomalies, correlated events, suspected services, and root-cause hypotheses.
- The user can ask questions such as “Why is service A suspected?”, “Show the timeline around the first error spike”, or “Compare this incident with previous samples.”
- The user can request visualizations such as error frequency over time, anomaly score distribution, service dependency event counts, or clustered log templates.
- Chart generation should run through a controlled Python visualization tool that accepts validated data slices and chart specs, produces static artifacts, and records the request in the audit log.
- Risky tool calls, data exports, or external actions should pause for explicit human approval.

## Security Model Emphasis

Security should be a core differentiator, not a checklist:

- **Prompt injection defense**: classify untrusted incident text, logs, ticket comments, and runbook snippets as data, not instructions.
- **Sensitive data redaction**: detect and mask tokens, secrets, IPs, emails, internal hostnames, and configurable patterns before LLM/tool exposure.
- **Permission boundaries**: define tools by capability and risk level; default to read-only analysis in V1.
- **Human approval**: require approval for risky actions such as exporting data, querying external systems, or proposing remediation commands.
- **Auditability**: persist who asked what, which tool ran, what data slice was used, what was redacted, and which evidence supported each hypothesis.
- **Explainability**: every answer should cite evidence IDs or timeline events rather than only producing natural-language conclusions.

## ML Scope Using scikit-learn

Use scikit-learn for practical, inspectable ML instead of opaque model-heavy features:

- `TfidfVectorizer` or similar text features for log message templates.
- `MiniBatchKMeans` or `KMeans` for log clustering and recurring pattern discovery.
- `IsolationForest`, `LocalOutlierFactor`, or `OneClassSVM` for anomaly detection on event counts, latency-like fields, or derived time-window features.
- Simple supervised classifiers only after labeled sample data exists.
- Model persistence should prefer safer formats such as `skops.io` over raw pickle when persistence is needed.

V1 should ship with seeded datasets and deterministic expected outputs so tests can verify anomaly ranking, clustering stability bounds, redaction behavior, and chart artifact generation.

## Local Infra Strategy with Podman

- Use Podman as the supported local runtime.
- Provide Podman-compatible compose instructions, avoiding Docker-specific assumptions.
- Start V1 with SQLite where possible to reduce setup friction; introduce Postgres via Podman only when session persistence, queryability, or portfolio credibility justifies it.
- Keep sample datasets mounted as local volumes or packaged fixtures.
- Document commands using `podman` / `podman compose` or a Podman-compatible compose workflow.

## Testable Milestones

1. **Project skeleton and tests**: create package structure, pytest setup, first failing tests, and CI-ready commands.
2. **Dataset ingestion**: parse seeded logs/events into normalized incident records with unit tests.
3. **Redaction and prompt-boundary layer**: prove sensitive data is masked and untrusted text cannot become tool instructions.
4. **Timeline and evidence store**: build incident timelines with stable evidence IDs and tests.
5. **scikit-learn analysis**: add anomaly scoring and log clustering with deterministic fixture expectations.
6. **Investigation API**: expose sessions, evidence, findings, and chart artifact endpoints.
7. **Controlled chart generation**: generate tested Python chart artifacts from validated data slices.
8. **Minimal dashboard**: show timeline, findings, evidence, charts, and audit log without Grafana/Prometheus.
9. **Agent interaction loop**: add Q&A over investigation state with evidence-cited answers.
10. **LangGraph orchestration**: introduce a small graph only after core services are stable; include human approval interrupts for risky actions.

## Main Risks and Simplifications

- **Risk: overengineering V1**. Simplification: ship one strong local incident workflow before integrations.
- **Risk: LangGraph sprawl**. Simplification: keep domain logic in services; use graph nodes as orchestration wrappers.
- **Risk: weak ML credibility**. Simplification: use small, explainable scikit-learn models with visible features and tested outputs.
- **Risk: dashboard scope creep**. Simplification: build only timeline, evidence cards, hypotheses, audit log, and a few high-value charts.
- **Risk: security claims without proof**. Simplification: write explicit tests for redaction, prompt injection handling, permission checks, and audit records.
- **Risk: local infra friction**. Simplification: SQLite-first, Podman-only optional services, and seeded fixtures.

Yes. The proposal should frame DAFI Sentinel as a security-first AI incident investigation workbench with a testable V1: local seeded data, scikit-learn analysis, controlled chart generation, product-owned dashboard, and a small LangGraph-backed interaction flow introduced after deterministic services are in place.
